**A. Context bridge and narrative structure (add at the top of ## Execution)**

- Use protocol-specific terminology (counterexample, obligation gate, scope narrowing, etc.) with inline explanation on first use; subsequent uses are bare
- Output length scales with protocol complexity — adversarial protocols produce longer output than evaluative ones because they have more phases and challenge-rebuttal cycles
- Never dump schema fields directly — the output is narrative prose for humans, not structured data

**B. Adversarial protocol family execution instructions**

Add a subsection: `### Adversarial protocols (CFFP, CDP, CBP, HEP, ATP, EMP)`

Instruct Claude to read the selected protocol's `.opt.cue` file (path already determined by routing), then execute its phases in order. The instructions must be schema-directed — telling Claude to follow what the loaded schema defines, not hard-coding phase sequences. Specifically:

**Phase 1:** Establish starting conditions from the protocol's Phase1 type (invariants, phenomenon, term, source construct, composed forms, or subject — depending on which protocol was routed). Apply eager gate: if the schema requires "at least one X" (the `[_, ...]` constraint pattern) and none can be established from the user's problem, stop here with a gate diagnosis and suggestions.

**Phase 2:** Generate candidates/hypotheses per the protocol's Phase2 type. Apply eager gate: if at least one candidate cannot be formulated, stop with a gate diagnosis.

**Phase 3:** Apply adversarial pressure. For each candidate, generate challenges of the types defined in the protocol's Phase3 schema (these vary per protocol — CFFP uses #Counterexample and #CompositionFailure, HEP uses #EvidenceRebuttal, etc.). For each challenge, evaluate whether a rebuttal holds (rebuttal kinds: refutation, scope_narrowing; some challenge types are not rebuttable like CompositionFailure). Compute derived: eliminated list (with elimination_reason and source_challenge_id) + survivors list (with any scope_narrowings from valid scope_narrowing rebuttals).

**If derived.survivors is empty → Phase 3b (revision loop):**
- Summarize the failed first pass briefly: "All N candidates were eliminated because [what pressure they couldn't survive]."
- Identify the diagnosis label from the protocol's Phase3b diagnosis enum.
- Check for skip-retry diagnoses: if diagnosis is `construct_incoherent` (CFFP), `construct_not_decomposable` (CDP), or `transfer_not_viable` (ATP), skip retry entirely. Report with the reframe_and_close resolution and explain why retrying won't help. Include suggestions for how the user could reframe. Stop.
- Otherwise: state diagnosis label explicitly in output ("Revision triggered: **[diagnosis]** — [what this means and what's being revised]"). Apply the resolution enum value. Re-run Phases 2-3 in full with revised parameters (show the full second pass output).
- If the second pass also produces empty survivors: report and stop. No infinite loops.
- Note: HEP tracks `revision_count: uint` and CDP has `max_revisions: uint` — respect these schema-declared limits.

**If derived.survivors has exactly 1:** Skip Phase 4. Proceed to Phase 5 with the single survivor.

**If derived.survivors > 1 → Phase 4:** Select one survivor with explicit rationale per the protocol's Phase4 type. (HEP Phase4 has two branches: single_survivor does confidence assessment, multi-survivor attempts discriminating experiments.)

**Phase 5 (obligation gate):**
Evaluate all obligations defined in the protocol's Phase5 schema. The gate field varies:
- CFFP: `all_provable` (using #StaticObligation with `provable: bool`)
- CDP: `all_ready` (using #PartReadiness with `ready: bool`)
- All others: `all_satisfied` (using protocol-specific obligation types)

If the gate fails: show all completed phases in full (the journey matters per locked decision), then display a clear gate diagnosis stating which obligation failed and why, plus actionable suggestions for how the user could reframe or modify input. Hard stop — no workarounds, no forcing past unmet gates.

**Phase 6** (only when Phase 5 gate passes): Produce the canonical form / adopted explanation / validated transfer / authorized parts / emergence map — per the protocol's Phase6 type.

Special note for CDP: Phase6 produces `cffp_instructions` — mention in the conclusion that CDP has authorized CFFP runs for each part, and the user can invoke `/socrates` with the part description to begin formalization.

**C. Obligation gate eager enforcement summary**

Add a brief instruction block reminding Claude of the eager enforcement principle: check preconditions at EVERY phase transition, not just at the schema's explicit Phase5 gate. If Phase 1 requires at least one invariant and zero are established, stop before Phase 2. If Phase 2 requires at least one candidate and zero are generated, stop before Phase 3. The Phase5 gate is the formal obligation gate, but the eager principle means Claude validates phase preconditions throughout.

**Line budget:** The entire Execution section after this task should be approximately 80-100 lines. Be instruction-dense — one sentence per instruction, no explanatory filler. The .opt.cue schemas provide the detail; SKILL.md provides the execution sequence and behavioral rules.
