Add two more subsections to the `## Execution` section after the evaluative protocol subsection:

**A. `### Exploratory protocol (ADP)`**

Instruct Claude on ADP's execution model, which is entirely different from adversarial and evaluative families:

1. Establish the subject type from the user's problem: new_construct, new_domain, breaking_change, or decision (per ADP's #ADPSubject).
2. Declare the design constraints that apply to this run (from ADP's #DesignConstraint types).
3. Execute rounds. Each round has a type (probe, pressure, synthesis, handoff) and involves 5-6 personas (formalist, implementor, adversary, operator, consumer, referee). For each round:
   - Narrate what each persona argues in THIRD PERSON. Example: "The formalist raises a decidability concern, arguing that..." Do NOT write as first-person dialogue ("As the formalist, I believe...") and do NOT write as bare transcript ("Formalist: ..."). The narrative voice describes each persona's contribution.
   - After each round, the referee applies constraint checks to all proposals. If a proposal fails a constraint check (`#ConstraintCheckSet.passed: false`), it is blocked from proceeding.
4. Typical structure: probe round → pressure round → synthesis round. If synthesis does not converge, additional pressure/synthesis rounds may occur, or the referee declares scope_reduction.
5. Referee declares the terminal outcome: `design_mapped` (produces #DesignMap), `exhaustion` (rounds exhausted without convergence), or `scope_reduction` (problem narrowed to a subset).
6. If `design_mapped`: render the DesignMap contents as the conclusion. Note any constructs marked as CFFP-ready (candidates for formalization via /socrates).

**Line budget for ADP subsection:** approximately 15-20 lines.

**B. `### Multi-protocol handoff`**

Add instructions for composite protocol sequences (per locked decisions from CONTEXT.md):

1. After the first protocol completes in a composite sequence, show an explicit handoff section before beginning the second protocol. Format: "[First protocol] established that [key output]. This feeds into [second protocol] as follows: [explicit input mapping]."

2. For OVP → HEP specifically (the confirmed prerequisite composite): extract OVP's Phase4 `validated_observation` (phenomenon, confidence, caveats). Use this as HEP's `#Phenomenon.observation` input. Incorporate OVP's caveats as `known_exclusions` in HEP's initial hypothesis set. Do NOT re-derive the phenomenon from the user's original problem description — use OVP's refined output.

3. Early termination: if the first protocol's outcome invalidates the need for the second, explain why the sequence ends, show the first protocol's complete output, and suggest alternatives. Example: if OVP returns `artifact` (the phenomenon is not real), HEP would operate on a false premise — stop and explain.

4. No redundant routing block when the second protocol begins — the initial composite routing overview (displayed before the first protocol) is sufficient.

5. For any future composite sequences beyond OVP→HEP: follow the same handoff pattern. Check the first protocol's terminal output type and map relevant fields to the second protocol's Phase1 input type.
