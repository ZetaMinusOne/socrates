  <action>
    **Part A: Create the stripping script**

    Create `.claude/skills/socrates/scripts/strip_cue.py` — a Python 3 script (~40-80 lines) that:

    1. Takes the skill directory as the working base (defaults to `..` relative to script location, i.e., `.claude/skills/socrates/`)
    2. Defines the file mapping from raw submodule paths to optimized output paths:
       - `dialectics/dialectics.cue` → `protocols/dialectics.opt.cue`
       - `dialectics/governance/routing.cue` → `protocols/routing.opt.cue`
       - `dialectics/protocols/adversarial/{name}.cue` → `protocols/adversarial/{name}.opt.cue`
       - `dialectics/protocols/evaluative/{name}.cue` → `protocols/evaluative/{name}.opt.cue`
       - `dialectics/protocols/exploratory/adp.cue` → `protocols/exploratory/adp.opt.cue`
    3. For each file, applies these stripping rules (in order):
       a. **Remove block comment sections:** Multi-line `//` comment blocks (3+ consecutive comment-only lines) that explain design philosophy, rationale, or history for external readers. These are documentation, not execution semantics.
       b. **Remove divider lines:** Lines matching `// ─` or `// ===` or `// ---` patterns (visual formatting dividers).
       c. **Collapse multiple blank lines** to a single blank line.
       d. **PRESERVE:** All CUE structure (types, fields, constraints, enums), all inline comments that describe field semantics (e.g., `// true if this rebuttal holds under adversarial review`), all phase sequence definitions, all `#ObligationGate` and `#RevisionLoop` references with their semantic explanations.
    4. The stripping heuristic for distinguishing "block documentation comments" from "inline semantic comments":
       - A **block comment** is 3+ consecutive lines where each line starts with `//` (possibly with leading whitespace) and there is no CUE code on any of those lines. Strip these blocks entirely.
       - An **inline comment** is a `//` on the same line as CUE code, or a single-line `//` comment immediately preceding a CUE field/type definition. Preserve these.
       - A **short comment group** (1-2 consecutive `//` lines before a CUE definition) is preserved — these are field/type semantic descriptions.
    5. Creates output directories as needed (`protocols/adversarial/`, `protocols/evaluative/`, `protocols/exploratory/`).
    6. Prints a summary: filename, raw size, stripped size, reduction percentage.
    7. Warns if any stripped file exceeds 16,000 characters.
    8. NOTE on `recording.cue`: Do NOT strip `dialectics/governance/recording.cue` — it is only needed in Phase 4 (`--record` flag) and is not referenced in Phase 1 SKILL.md file list. It can be added later if needed.

    The script should be idempotent — running it again overwrites existing `.opt.cue` files.

    **Part B: Run the stripping script**

    Execute: `python3 .claude/skills/socrates/scripts/strip_cue.py`

    This generates all 15 `.opt.cue` files into the `protocols/` directory tree.

    **Part C: Validate the output**

    For each generated `.opt.cue` file:
    1. Confirm it exists and is non-empty
    2. Confirm it is under 16,000 characters
    3. Confirm it retains CUE structure (has `#` type definitions or field definitions)
    4. Confirm the size reduction is in the 30-70% range (if less than 30% reduction, the stripping may be too conservative; if more than 70%, it may be too aggressive)

    If any file exceeds 16,000 characters after stripping, manually review and identify additional comment blocks that can be safely removed while preserving execution semantics. Adjust the script or the specific file as needed.

    **Stripping criteria (from CONTEXT.md locked decision):**
    - Priority: execution fidelity — keep everything Claude needs to faithfully execute protocols
    - Preserve: field descriptions, constraints, phase sequences, obligation gate semantics
    - Strip: block-comment documentation sections (design philosophy/rationale), divider lines, excessive blank lines
    - Test: A reader of the stripped file who hasn't seen the original should understand what each field means and how each phase works
