  <files>.claude/skills/socrates/SKILL.md</files>
  <action>
Append a new `## Output Rendering` section at the end of SKILL.md (after the existing `## Execution` section and its subsections). This section governs what happens after protocol execution completes (or fails at a gate).

The section must contain these subsections and instructions:

**Main rendering branch:**
- If no flags detected: use existing narrative prose behavior (no change — this is the default). No additional instructions needed for this path.
- If `--structured` detected: render structured JSON output (see below).
- If `--record` detected: render #Record JSON output (see below).
- If both detected: render combined JSON object `{"structured": {protocol output envelope}, "record": {#Record}}`.

**Structured output format (`--structured`):**
Output ONLY a JSON object. No preamble text, no trailing explanation. Use a single markdown code fence (```json ... ```) as the formatting wrapper.

Envelope structure:
```json
{"protocol": "{ACRONYM}", "routed_via": "{structural_feature}", "output": {full protocol instance}}
```

The `output` field contains ALL executed phases as JSON objects — every phase that was executed appears, including Phase 3b if triggered, Phase 4 if survivors > 1. The field names and nesting follow the protocol's `#ProtocolInstance` type from its `.opt.cue` file.

Challenge/rebuttal narrative text is included as string `description` fields within the JSON objects (verbose but complete audit trail per locked decision).

Revision loops include both passes: `[{"pass": 1, "outcome": "all_eliminated", ...}, {"pass": 2, "outcome": "survivors", ...}]`.

Evaluative protocols use the same full-phase pattern for consistency.

**Gate failure format:**
When execution stops at a gate in structured mode, output:
```json
{"outcome": "gate_failed", "gate": "{gate description}", "completed_phases": [...], "suggestions": [...]}
```
No envelope wrapper — gate failures replace the normal output entirely.

**Composite sequence format:**
When routing produces a composite sequence and `--structured` is active:
```json
{"sequence": [{"protocol": "...", "routed_via": "...", "output": {...}}, ...]}
```
Early termination appends: `"early_termination": {"reason": "...", "stopped_at": "...", "suggested_next": "..."}`.

**Record output format (`--record`):**
Read `dialectics/governance/recording.cue` (already loaded in Flag Handling if --record detected). Project the completed run into a `#Record` JSON object.

Field population rules:
- `record_id`: Format `"rec-{protocol_lower}-{YYYYMMDD}-{4-char-hex}"` (e.g., `"rec-cffp-20260228-a7f3"`)
- `source_run.run_id`: Format `"{protocol_lower}-{YYYYMMDD}-{4-char-hex}"` (same suffix as record_id, without "rec-" prefix)
- `source_run.protocol`: Protocol acronym
- `source_run.run_version`: From the protocol's `#Protocol.version` field in the loaded .opt.cue
- `source_run.subject`: The user's problem text (after flag stripping)
- `source_run.started` and `completed`: ISO 8601 using today's date, `T00:00:00Z` time portion
- `dispute.kind`: Map from structural feature using this table:
  - `competing_candidates` → `"candidate_selection"`
  - `term_inconsistency` → `"term_ambiguity"`
  - `argument_fragility` → `"assumption_audit"`
  - `unknown_design_space` → `"design_mapping"`
  - `construct_incoherence` → `"construct_repair"`
  - `implementation_gap` → `"implementation_check"`
  - `revision_pressure` / `deprecation_pressure` → `"governance_case"`
  - `cross_run_conflict` → `"cross_run_conflict"`
  - `structural_transfer` → `"analogy_transfer"`
  - `composition_emergence` → `"composition_emergence"`
  - `observation_validity` → `"observation_validity"`
  - `resource_constrained_choice` → `"prioritization"`
  - `causal_ambiguity` → `"candidate_selection"` (note in dispute.description that this is causal hypothesis elimination)
- `dispute.prior_runs`: Default `[]`
- `resolution.status`: `"decided"` if canonical/adopted/validated, `"open"` if indeterminate, `"rejected"` if all eliminated
- `resolution.eliminated_count` and `survivors`: From derived phase data
- `acknowledged_limitations`: From scope_narrowing rebuttals
- `dependencies.consumed` and `produced`: Default `[]`
- `tags`: Auto-generated array from protocol category, protocol acronym lowercase, and resolution status (e.g., `["adversarial", "cffp", "decided"]`)
- `next_actions`: Populated ONLY when the protocol explicitly names a follow-up (CDP → CFFP instructions, RCP blocked → CBP needed). No speculative suggestions.
- `notes`: Default `""`

**Ambiguous/unroutable routing with flags:**
- Ambiguous routing with `--structured`: Ask the clarifying question in plain text (not JSON). Structured output requires a selected protocol. After clarification and successful re-routing, apply the flag to the result.
- Unroutable routing with `--structured`: Return `{"outcome": "unroutable", "message": "..."}`.
- Same behavior applies with `--record` flag.

Target ~50-65 lines for the entire Output Rendering section. SKILL.md total should land at ~310-330 lines after both tasks.
